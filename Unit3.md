### **AAP-1: Data Representation and Organization**
* **AAP-1.A.1** A variable is an abstraction inside a program that can hold a value.
* **AAP-1.A.2** Using meaningful variable names helps readability and understanding.
* **AAP-1.A.3** Some languages provide types (numbers, Booleans, lists, strings).
* **AAP-1.A.4** Some values are better suited to one type than another.
* **AAP-1.B.1** Assignment operator changes a variable’s value.
* **AAP-1.B.2** Assignment operator `←` assigns evaluated result to variable.
* **AAP-1.B.3** A variable stores its most recent assigned value.
* **AAP-1.C.1** A list is an ordered sequence of elements.
* **AAP-1.C.2** An element is a value in a list with a unique index.
* **AAP-1.C.3** An index references list/string elements using natural numbers.
* **AAP-1.C.4** A string is an ordered sequence of characters.
* **AAP-1.D.1** Data abstraction separates abstract properties from representation.
* **AAP-1.D.2** Data abstractions manage complexity by naming collections of data.
* **AAP-1.D.3** Lists can be used to create data abstractions.
* **AAP-1.D.4** Data abstraction can make programs easier to develop/maintain.
* **AAP-1.D.5** Data abstractions may contain different types of elements.
* **AAP-1.D.6** Lists allow related items to be treated as a single value.
* **AAP-1.D.7** Notation `[value1, value2, ...]` creates a list.
* **AAP-1.D.8** List indices are 1-based; out-of-range indices cause errors.

### **AAP-2: Sequencing, Selection, Iteration**
* **AAP-2.A.1** An algorithm is a finite set of instructions for a task.
* **AAP-2.A.2** Algorithms can be expressed in natural language, diagrams, pseudocode.
* **AAP-2.A.3** Algorithms executed by programs are implemented in languages.
* **AAP-2.A.4** Every algorithm uses sequencing, selection, iteration.
* **AAP-2.B.1** Sequencing applies steps in order given.
* **AAP-2.B.2** A code statement expresses an action.
* **AAP-2.B.3** An expression can be value, variable, operator, or procedure call.
* **AAP-2.B.4** Expressions evaluate to a single value.
* **AAP-2.B.5** Expression evaluation follows defined order of operations.
* **AAP-2.B.6** Sequential statements run in order.
* **AAP-2.B.7** Clarity/readability matter when expressing algorithms.
* **AAP-2.C.1** Arithmetic operators include +, -, \*, /, MOD.
* **AAP-2.C.2** `a MOD b` gives the remainder.
* **AAP-2.C.3** Operators include +, -, \*, /, MOD.
* **AAP-2.C.4** Standard order of operations applies.
* **AAP-2.D.1** String concatenation joins strings end-to-end.
* **AAP-2.D.2** A substring is part of an existing string.
* **AAP-2.E.1** A Boolean is true or false.
* **AAP-2.E.2** Relational operators (=, ≠, >, <, ≥, ≤) evaluate to Boolean values.
* **AAP-2.F.1** Logical operators NOT, AND, OR evaluate to Boolean values.
* **AAP-2.F.2** `NOT condition` inverts Boolean.
* **AAP-2.F.3** `condition1 AND condition2` is true if both are true.
* **AAP-2.F.4** `condition1 OR condition2` is true if either/both are true.
* **AAP-2.F.5** Operands for logical operators must be Boolean expressions/values.
* **AAP-2.G.1** Selection executes code based on condition truth.
* **AAP-2.H.1** If-statements affect flow by executing code based on condition.
* **AAP-2.H.2** `IF(condition) { statements }` executes if true.
* **AAP-2.H.3** `IF ... ELSE ...` executes one block or the other.
* **AAP-2.I.1** Nested conditionals are conditionals inside conditionals.
* **AAP-2.J.1** Iteration repeats instructions a number of times or until condition met.
* **AAP-2.K.1** Iteration changes flow by repeating statements until stop condition.
* **AAP-2.K.2** `REPEAT n TIMES {}` executes n times.
* **AAP-2.K.3** `REPEAT UNTIL(condition)` repeats until true.
* **AAP-2.K.4** Infinite loop occurs if condition never true.
* **AAP-2.K.5** If condition true initially, loop body isn’t executed.
* **AAP-2.L.1** Algorithms can be written differently but achieve same task.
* **AAP-2.L.2** Similar algorithms can yield different results/side effects.
* **AAP-2.L.3** Some conditionals can be written as Boolean expressions.
* **AAP-2.L.4** Some Boolean expressions can be written as conditionals.
* **AAP-2.L.5** Different algorithms can solve same problem.
* **AAP-2.M.1** Algorithms can be created, combined, or modified.
* **AAP-2.M.2** Knowledge of existing algorithms helps construct new ones.
* **AAP-2.M.3** Reusing existing algorithms reduces time, errors, and complexity.
* **AAP-2.N.1** List operations: indexing, assigning, inserting, appending, removing, length.
* **AAP-2.N.2** List procedures follow language syntax rules.
* **AAP-2.O.1** List traversal can be complete or partial.
* **AAP-2.O.2** Iteration statements traverse lists.
* **AAP-2.O.3** `FOR EACH item IN aList {}` iterates list.
* **AAP-2.O.4** Existing list algorithms: min, max, sum, average.
* **AAP-2.O.5** Linear search checks each element until found or exhausted.
* **AAP-2.P.1** Binary search starts in middle, halves data each step.
* **AAP-2.P.2** Data must be sorted for binary search.
* **AAP-2.P.3** Binary search is often more efficient than linear search.

### **AAP-3: Procedures, Libraries, and Simulations**
* **AAP-3.A.1** A procedure is a named group of instructions, may have parameters/returns.
* **AAP-3.A.2** Procedures are also called methods/functions.
* **AAP-3.A.3** Parameters are input variables; arguments assign values.
* **AAP-3.A.4** A procedure call interrupts execution, executes procedure, then returns.
* **AAP-3.A.5** `procName(arg1, arg2, …)` calls procedures.
* **AAP-3.A.6** `DISPLAY(expression)` outputs expression.
* **AAP-3.A.7** `RETURN(expression)` returns value to caller.
* **AAP-3.A.8** `result ← procName(args)` assigns procedure result.
* **AAP-3.A.9** `INPUT()` accepts user input.
* **AAP-3.B.1** Procedural abstraction provides a name for a process.
* **AAP-3.B.2** Abstraction allows breaking problems into subproblems.
* **AAP-3.B.3** Subdividing programs into subprograms is modularity.
* **AAP-3.B.4** Abstraction generalizes functionality and reduces duplicate code.
* **AAP-3.B.5** Parameters generalize procedures for reuse.
* **AAP-3.B.6** Abstraction improves readability.
* **AAP-3.B.7** Abstraction allows internal changes without affecting users.
* **AAP-3.C.1** Procedures defined as `PROCEDURE name(params) { statements }`.
* **AAP-3.C.2** Procedures can return values with `RETURN(expression)`.
* **AAP-3.D.1** A software library contains reusable procedures.
* **AAP-3.D.2** Code segments can come from libraries or previous code.
* **AAP-3.D.3** Libraries simplify creating complex programs.
* **AAP-3.D.4** APIs specify how library procedures behave.
* **AAP-3.D.5** API/library documentation explains behavior and usage.
* **AAP-3.E.1** `RANDOM(a, b)` generates integer between a and b.
* **AAP-3.E.2** Random generation can produce different results on each run.
* **AAP-3.F.1** Simulations are abstractions of complex phenomena.
* **AAP-3.F.2** A simulation represents state with varying values.
* **AAP-3.F.3** Simulations mimic real-world events to investigate phenomena.
* **AAP-3.F.4** Simulations simplify/remove details to abstract.
* **AAP-3.F.5** Simulations may contain bias in what’s included/excluded.
* **AAP-3.F.6** Simulations are useful when real experiments are impractical.
* **AAP-3.F.7** Simulations help form/refine hypotheses.
* **AAP-3.F.8** Random numbers simulate real-world variability.

### **AAP-4: Algorithmic Efficiency and Undecidability**
* **AAP-4.A.1** A problem is a task that may/may not be solvable algorithmically.
* **AAP-4.A.2** Decision problems have yes/no answers; optimization finds best solution.
* **AAP-4.A.3** Efficiency estimates computational resources used by algorithm.
* **AAP-4.A.4** Efficiency can be determined mathematically.
* **AAP-4.A.5** Efficiency can be measured by counting executions.
* **AAP-4.A.6** Different correct algorithms for same problem can have different efficiency.
* **AAP-4.A.7** Polynomial time = reasonable; exponential/factorial = unreasonable.
* **AAP-4.A.8** Some problems lack efficient algorithms; approximations are used.
* **AAP-4.A.9** Heuristics give “good enough” solutions when optimal isn’t practical.
* **AAP-4.B.1** A decidable problem has an algorithm producing correct outputs for all inputs.
* **AAP-4.B.2** An undecidable problem has no algorithm guaranteed to solve all cases.
* **AAP-4.B.3** Some undecidable problems have solvable instances but not all.
