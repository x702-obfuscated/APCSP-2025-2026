### **Topic 1.1 – Collaboration**

* **CRD-1.A.1** A computing innovation includes a program as an integral part of its function.
* **CRD-1.A.2** A computing innovation can be physical (e.g., self-driving car), nonphysical software (e.g., picture editing software), or a nonphysical computing concept (e.g., e-commerce).
* **CRD-1.A.3** Effective collaboration produces a computing innovation that reflects diverse talents and perspectives.
* **CRD-1.A.4** Collaboration with diverse perspectives helps avoid bias in computing innovations.
* **CRD-1.A.5** Consultation and communication with users are important in developing computing innovations.
* **CRD-1.A.6** Information from potential users helps understand a program’s purpose from diverse perspectives.
* **CRD-1.B.1** Online tools support collaboration by allowing sharing and feedback on ideas and documents.
* **CRD-1.B.2** Common models such as pair programming facilitate collaboration.
* **CRD-1.C.1** Effective collaborative teams practice interpersonal skills like communication, consensus building, conflict resolution, and negotiation.

### **Topic 1.2 – Program Function and Purpose**

* **CRD-2.A.1** The purpose of computing innovations is to solve problems or pursue interests through creative expression.
* **CRD-2.A.2** Understanding purpose improves developers’ ability to create computing innovations.
* **CRD-2.B.1** A program is a collection of program statements that performs a specific task when run.
* **CRD-2.B.2** A code segment is a collection of statements that is part of a program.
* **CRD-2.B.3** Programs must work for a variety of inputs and situations.
* **CRD-2.B.4** Program behavior is how it functions during execution and is often described by user interaction.
* **CRD-2.B.5** A program can be described broadly by what it does or in detail by how statements accomplish it.
* **CRD-2.C.1** Program inputs are data sent to a computer for processing; inputs can be tactile, audio, visual, or text.
* **CRD-2.C.2** An event is associated with an action and supplies input data.
* **CRD-2.C.3** Events can be generated by actions like key presses, mouse clicks, or program starts.
* **CRD-2.C.4** Inputs usually affect outputs.
* **CRD-2.C.5** Event-driven programming executes program statements when triggered, not sequentially.
* **CRD-2.C.6** Input can come from users or other programs.
* **CRD-2.D.1** Program outputs are any data sent to a device and can be tactile, audio, visual, or text.
* **CRD-2.D.2** Output is usually based on a program’s input or prior state.

### **Topic 1.3 – Program Design and Development**

* **CRD-2.E.1** A development process can be ordered/intentional or exploratory.
* **CRD-2.E.2** Common phases: investigating, designing, prototyping, testing.
* **CRD-2.E.3** Iterative processes require refinement/revision based on feedback, testing, or reflection.
* **CRD-2.E.4** Incremental processes break problems into smaller pieces, ensuring each works before adding.
* **CRD-2.F.1** Program design incorporates investigation to determine requirements.
* **CRD-2.F.2** Investigation helps identify constraints and user concerns.
* **CRD-2.F.3** Investigation methods: surveys, user testing, interviews, direct observation.
* **CRD-2.F.4** Program requirements describe how it functions and user interactions.
* **CRD-2.F.5** A specification defines program requirements.
* **CRD-2.F.6** The design phase outlines how to meet the specification.
* **CRD-2.F.7** Design phase activities include brainstorming, planning, modular organization, UI diagrams, and testing strategies.
* **CRD-2.G.1** Program documentation describes functions of code and development.
* **CRD-2.G.2** Comments are documentation inside code; ignored during execution.
* **CRD-2.G.3** Programmers should document throughout development.
* **CRD-2.G.4** Documentation helps develop and maintain correct programs individually or collaboratively.
* **CRD-2.G.5** If comments aren’t supported, other documentation methods are needed.
* **CRD-2.H.1** Code segments developed collaboratively or from other sources should be acknowledged.
* **CRD-2.H.2** Acknowledgements should include origin or author’s name.

### **Topic 1.4 – Identifying and Correcting Errors**

* **CRD-2.I.1** A logic error is a mistake causing incorrect/unexpected behavior.
* **CRD-2.I.2** A syntax error occurs when programming language rules aren’t followed.
* **CRD-2.I.3** A run-time error occurs during execution; languages define their own.
* **CRD-2.I.4** An overflow error occurs when a number exceeds defined range.
* **CRD-2.I.5** Ways to find/correct errors: test cases, hand tracing, visualizations, debuggers, extra output statements.
* **CRD-2.J.1** Testing uses defined inputs to ensure expected outcomes; results are used to revise.
* **CRD-2.J.2** Defined test inputs should include values at/beyond extremes of input data.
* **CRD-2.J.3** Program requirements help identify appropriate test inputs.
